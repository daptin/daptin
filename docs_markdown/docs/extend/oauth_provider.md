# OAuth Provider

Daptin can act as an OAuth 2.0 authorization server for applications that need to sign users in through Daptin and receive OAuth access tokens.

This is different from OAuth Connections:

| Feature | Direction | Tables |
|---------|-----------|--------|
| OAuth Connections | Daptin consumes tokens from Google, GitHub, Dropbox, and other providers | `oauth_connect`, `oauth_token` |
| OAuth Provider | Daptin issues authorization codes, access tokens, refresh tokens, and OpenID Connect ID tokens to client apps | `oauth_app`, `oauth_code`, `oauth_access`, `oauth_refresh`, `oauth_key` |

## Supported flows

| Flow | Status |
|------|--------|
| Authorization Code | Supported |
| Refresh Token | Supported |
| PKCE S256 | Supported and required |
| PKCE plain | Supported for compatibility |
| OpenID Connect discovery | Supported |
| JWKS endpoint | Supported |
| UserInfo endpoint | Supported |
| Client Credentials | Not supported |
| Implicit flow | Not supported |

## Endpoints

| Endpoint | Method | Purpose |
|----------|--------|---------|
| `/.well-known/oauth-authorization-server` | GET | OAuth 2.0 authorization server metadata |
| `/.well-known/openid-configuration` | GET | OpenID Connect discovery metadata |
| `/oauth/authorize` | GET | Authorization code request |
| `/oauth/token` | POST | Authorization code and refresh token exchange |
| `/oauth/revoke` | POST | Token revocation |
| `/oauth/introspect` | POST | Token introspection |
| `/oauth/userinfo` | GET, POST | OpenID Connect user profile |
| `/oauth/jwks` | GET | JSON Web Key Set for ID token verification |

## Configure a client application

Register clients through the Daptin action API. `oauth_app` is the internal storage table and is not the client-facing management contract.

```bash
curl -X POST http://localhost:6336/action/oauth_app/register_client \
  -H "Authorization: Bearer $ADMIN_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "attributes": {
      "name": "Example Client",
      "redirect_uris": "https://client.example.com/callback",
      "scopes": "openid profile email",
      "grants": "authorization_code,refresh_token",
      "is_confidential": true
    }
  }'
```

The response includes generated credentials. `client_secret` is returned only at registration or rotation time.

```json
[
  {
    "ResponseType": "oauth_app",
    "Attributes": {
      "reference_id": "019dd82f...",
      "name": "Example Client",
      "client_id": "dapc_...",
      "client_secret": "daps_...",
      "redirect_uris": "https://client.example.com/callback",
      "scopes": "openid profile email",
      "grants": "authorization_code refresh_token",
      "is_confidential": true,
      "is_enabled": true,
      "secret_returned": true,
      "secret_store": "bcrypt"
    }
  }
]
```

### Client management actions

| Endpoint | Purpose |
|----------|---------|
| `POST /action/oauth_app/register_client` | Register a new client and return generated credentials |
| `POST /action/oauth_app/update_client` | Update allowed client metadata |
| `POST /action/oauth_app/rotate_client_secret` | Generate a new client secret and return it once |
| `POST /action/oauth_app/disable_client` | Disable a client |
| `POST /action/oauth_app/enable_client` | Re-enable a client |
| `POST /action/oauth_app/revoke_client_tokens` | Revoke active access and refresh tokens for a client |

Instance-bound actions use the app reference ID:

```json
{
  "attributes": {
    "oauth_app_id": "OAUTH_APP_REFERENCE_ID"
  }
}
```

### oauth_app fields

| Field | Description |
|-------|-------------|
| `name` | Human-readable client name |
| `client_id` | OAuth client identifier generated by Daptin |
| `client_secret` | Client secret generated by Daptin for confidential clients. Stored using Daptin password hashing behavior |
| `redirect_uris` | Allowed redirect URIs, separated by whitespace or commas |
| `scopes` | Allowed scopes, separated by whitespace or commas |
| `grants` | Allowed grants. Use `authorization_code,refresh_token` for the standard web flow |
| `is_confidential` | If true, token/revoke/introspect requests must authenticate with the client secret |
| `is_enabled` | If false, the client cannot be used |

## Authorization code flow

The user must already be signed in to Daptin. If not signed in, `/oauth/authorize` redirects to `oauth.login_url` from backend config, or `/auth/signin` by default.

### 1. Build a PKCE challenge

Generate a random verifier in the client application and derive an S256 challenge:

```text
code_challenge = BASE64URL(SHA256(code_verifier))
```

### 2. Send the user to Daptin

```text
GET http://localhost:6336/oauth/authorize?
  response_type=code&
  client_id=example-client&
  redirect_uri=https%3A%2F%2Fclient.example.com%2Fcallback&
  scope=openid%20profile%20email&
  state=random-csrf-state&
  code_challenge=BASE64URL_SHA256_VERIFIER&
  code_challenge_method=S256&
  nonce=random-oidc-nonce
```

Daptin validates the client, redirect URI, scopes, grant type, user session, and PKCE challenge before creating an authorization code.

On success, Daptin redirects to:

```text
https://client.example.com/callback?code=AUTHORIZATION_CODE&state=random-csrf-state
```

Invalid `redirect_uri` values are rejected directly with HTTP 400 and are not redirected.

### 3. Exchange the code for tokens

Confidential clients can use HTTP Basic authentication:

```bash
curl -X POST http://localhost:6336/oauth/token \
  -u "example-client:replace-with-a-long-random-secret" \
  -H "Content-Type: application/x-www-form-urlencoded" \
  -d "grant_type=authorization_code" \
  -d "code=$CODE" \
  -d "redirect_uri=https://client.example.com/callback" \
  -d "code_verifier=$CODE_VERIFIER"
```

Response:

```json
{
  "access_token": "opaque-access-token",
  "refresh_token": "opaque-refresh-token",
  "token_type": "Bearer",
  "expires_in": 3600,
  "scope": "openid profile email",
  "id_token": "eyJ..."
}
```

`id_token` is returned when the issued scope includes `openid`.

### 4. Use the access token

```bash
curl http://localhost:6336/oauth/userinfo \
  -H "Authorization: Bearer $ACCESS_TOKEN"
```

### 5. Refresh tokens

```bash
curl -X POST http://localhost:6336/oauth/token \
  -u "example-client:replace-with-a-long-random-secret" \
  -H "Content-Type: application/x-www-form-urlencoded" \
  -d "grant_type=refresh_token" \
  -d "refresh_token=$REFRESH_TOKEN"
```

Refresh tokens are rotated. The previous refresh token is revoked during a successful refresh and cannot be replayed.

## Token validation and revocation

### Introspect

```bash
curl -X POST http://localhost:6336/oauth/introspect \
  -u "example-client:replace-with-a-long-random-secret" \
  -H "Content-Type: application/x-www-form-urlencoded" \
  -d "token=$ACCESS_TOKEN"
```

Active response:

```json
{
  "active": true,
  "client_id": "example-client",
  "scope": "openid profile email",
  "token_type": "Bearer",
  "exp": 1777445590,
  "sub": "user-reference-id"
}
```

Inactive response:

```json
{
  "active": false
}
```

### Revoke

```bash
curl -X POST http://localhost:6336/oauth/revoke \
  -u "example-client:replace-with-a-long-random-secret" \
  -H "Content-Type: application/x-www-form-urlencoded" \
  -d "token=$ACCESS_TOKEN"
```

## OpenID Connect

Discovery is available at:

```text
GET /.well-known/openid-configuration
```

ID tokens are signed with RS256. Public keys are available at:

```text
GET /oauth/jwks
```

The signing key is created lazily the first time Daptin needs one, stored in `oauth_key`, and exposed through JWKS. The private key is stored encrypted.

## Configuration

| Config key | Type | Default | Purpose |
|------------|------|---------|---------|
| `oauth.issuer` | backend config | request scheme and host | Issuer advertised in discovery and used in ID tokens |
| `oauth.login_url` | backend config | `/auth/signin` | Login URL used when `/oauth/authorize` is called without a signed-in Daptin user |
| `encryption.secret` | backend config | instance default | Encrypts the OAuth signing private key |

Set config values as an administrator:

```bash
curl -X POST http://localhost:6336/_config/backend/oauth.issuer \
  -H "Authorization: Bearer $ADMIN_TOKEN" \
  -d "https://auth.example.com"
```

## Security behavior

- Authorization codes and bearer tokens are stored as SHA-256 hashes.
- Authorization codes expire after 10 minutes and can be used once.
- Access tokens expire after 1 hour.
- Refresh tokens expire after 30 days and rotate on every refresh.
- PKCE is required for authorization code issuance.
- Client IDs and client secrets are generated server-side by the registration/rotation actions.
- Client secrets are checked only for confidential clients.
- Token responses include `Cache-Control: no-store` and `Pragma: no-cache`.
- Invalid redirect URIs are rejected without redirecting.

Use HTTPS in production. OAuth bearer tokens must not be sent over plaintext HTTP outside local development.
